# AGENTS.md

## Repo Shape
- This is a personal Neovim Lua config, not an app/library repo; there are no project tests, CI workflows, or package manifests besides `lazy-lock.json`.
- `init.lua` is the entrypoint. It sets core options/keymaps, then loads `lua/config/lazy.lua`, `lua/config/special_keymaps.lua`, and `lua/config/snippets.lua`.
- Plugin specs live in `lua/config/plugins/*.lua` and are imported by Lazy via `{ import = "config.plugins" }`; add new plugin config there unless it truly belongs in `init.lua`.

## Commands
- Smoke-test config loading with `nvim --headless -u init.lua '+qa'` from the repo root.
- Plugin installation/update is managed by Lazy: use `:Lazy` interactively, `:Lazy update` for lockfile changes, or `nvim --headless '+Lazy! sync' '+qa'` for a headless sync.
- Expect first-load side effects: `config.lazy` bootstraps `lazy.nvim` if missing, `mason-tool-installer` has `run_on_start = true`, and Treesitter calls `require("nvim-treesitter").install(filetypes)` during startup.

## Neovim/API Assumptions
- This config uses modern Neovim APIs such as `vim.lsp.config`, `vim.lsp.enable`, `vim.hl`, and `vim.o.winborder`; do not rewrite LSP setup to legacy `lspconfig.SERVER.setup(...)` style.
- LSP capabilities come from `require("blink.cmp").get_lsp_capabilities()` in `lua/config/plugins/lsp.lua`; keep new server config compatible with Blink.
- Java is special-cased: `lua/config/plugins/nvim-java.lua` calls `require("java").setup()` and then `vim.lsp.enable("jdtls")`.

## Formatting And Tools
- Formatting-on-save is configured through Conform in `lua/config/plugins/conform.lua`; Lua uses `stylua`, Python uses `ruff_format`, JSON uses `jq`, Markdown uses `prettier`, and SQL uses `sqlfluff`.
- Mason tool names and LSP server names differ in places; update both `mason-tool-installer.lua` and `mason-lspconfig.lua` when adding managed tooling.
- `lazy-lock.json` is generated by Lazy; only edit it through Lazy commands unless the user explicitly asks for manual pin changes.
